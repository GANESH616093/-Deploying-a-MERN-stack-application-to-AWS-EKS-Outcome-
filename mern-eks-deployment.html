01 · Brief

Why EKS, and what's actually running

The app itself is a plain MERN to-do/inventory tool — a React SPA talking to an Express REST API, backed by MongoDB. The interesting part was never the app; it was learning to reason about it as a distributed system: what happens when a pod dies mid-request, how traffic actually reaches a container, and what "production" means once you're the one holding the pager.

Goal for this build was deliberately narrow: run all three tiers as independently scalable Kubernetes workloads on a real AWS account, with no single point of manual intervention between a git push and traffic serving the new version.

02 · Architecture

The cluster, laid out

One VPC, two availability zones, three subnets tiers. The load balancer and NAT gateways sit in public subnets; everything that runs application code sits in private ones. Nothing in the node group has a public IP.

03 · Build log

Seven steps, in the order they actually happened

Kept roughly chronological because the order matters — you can't attach an Ingress to a controller that doesn't exist yet, and you can't push images the cluster has no permission to pull.

01

Containerize the app

Multi-stage Dockerfile for the React client — build with Node, serve the static output through nginx — and a slim node:20-alpine image for the Express API. Confirmed both worked together locally with docker-compose before touching AWS.

# client/Dockerfile — build stage → nginx serve stage
docker build -t mern-client:1.0 ./client
docker build -t mern-api:1.0 ./server
docker compose up   # sanity check, local only
02

Push images to ECR

Created one repository per service, authenticated the Docker CLI against ECR, tagged with both a version and the git SHA so a rollback is always a known-good tag away.

aws ecr get-login-password --region ap-south-1 \
  | docker login --username AWS --password-stdin 111111111111.dkr.ecr.ap-south-1.amazonaws.com
docker tag mern-api:1.0 111111111111.dkr.ecr.ap-south-1.amazonaws.com/mern-api:a3f9c1
docker push 111111111111.dkr.ecr.ap-south-1.amazonaws.com/mern-api:a3f9c1
03

Provision the cluster

Used eksctl over the raw console — declarative, and it produces a VPC with public/private subnets across two AZs without hand-wiring route tables. Managed node group so AWS handles the underlying EC2 lifecycle.

eksctl create cluster \
  --name mern-prod --region ap-south-1 --version 1.29 \
  --nodegroup-name workers --node-type t3.medium \
  --nodes 3 --nodes-min 2 --nodes-max 5 --managed
04

Wire up networking

Associated an IAM OIDC provider with the cluster so pods can assume IAM roles directly (IRSA), then installed the AWS Load Balancer Controller through Helm — this is what turns a Kubernetes Ingress object into an actual ALB.

eksctl utils associate-iam-oidc-provider --cluster mern-prod --approve
helm install aws-load-balancer-controller eks/aws-load-balancer-controller \
  -n kube-system --set clusterName=mern-prod
05

Deploy the three workloads

Applied Deployments and ClusterIP Services for the frontend and API, and a StatefulSet for MongoDB with a PersistentVolumeClaim so data survives pod restarts. Connection string and JWT secret live in a Kubernetes Secret, not the image.

kubectl apply -f k8s/mongo-statefulset.yaml
kubectl apply -f k8s/api-deployment.yaml
kubectl apply -f k8s/client-deployment.yaml
kubectl get pods -w
06

Expose it through Ingress

Single Ingress resource, annotated for an internet-facing ALB, routing /api/* to the Express service and everything else to the React service. One load balancer for the whole app instead of one per service.

kubectl apply -f k8s/ingress.yaml
kubectl get ingress mern-ingress   # wait for ADDRESS to populate
07

Add the things that make it "production"

Liveness and readiness probes so the Service stops sending traffic to a pod before it's actually ready. A HorizontalPodAutoscaler on the API. A PodDisruptionBudget so node upgrades can't take out every API pod at once.

kubectl autoscale deployment express-api --cpu-percent=65 --min=3 --max=8
kubectl apply -f k8s/pdb.yaml
04 · Manifests

Two files worth showing

The API Deployment and the Ingress that fronts everything — trimmed for space, but this is the real shape of what's applied.

k8s/api-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: express-api
spec:
  replicas: 3
  strategy:
    rollingUpdate:
      maxUnavailable: 0
      maxSurge: 1
  template:
    spec:
      containers:
        - name: express-api
          image: ...ecr.../mern-api:a3f9c1
          envFrom:
            - secretRef:
                name: mongo-uri
          readinessProbe:
            httpGet: # /healthz, 5s interval
              path: /healthz
              port: 5000
k8s/ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: mern-ingress
  annotations:
    alb.ingress.kubernetes.io/scheme: internet-facing
    alb.ingress.kubernetes.io/target-type: ip
spec:
  rules:
    - http:
        paths:
          - path: /api
            backend:
              service:
                name: express-api
          - path: /
            backend:
              service:
                name: react-frontend # catch-all
05 · Field notes

What actually went wrong

Not in order — these surfaced at different points and got fixed as they came up.

storage

MongoDB needed to survive a pod restart

A plain Deployment loses its filesystem on every reschedule. Switched to a StatefulSet with a volumeClaimTemplate backed by an EBS gp3 volume, so the same disk reattaches to the same pod identity.

secrets

Connection strings didn't belong in Git

Started with plain Kubernetes Secrets, then moved to AWS Secrets Manager synced in through the External Secrets Operator — one less base64 string sitting in a YAML file.

rollout

Rolling updates were briefly dropping connections

Default readiness probe fired too early. Added a real /healthz endpoint that checks the Mongo connection, and set maxUnavailable: 0 so old pods don't terminate before new ones are ready.

cost

Three always-on nodes was overkill after hours

Traffic is near-zero overnight in this timezone. Set the node group's autoscaler floor to 2 and let the HPA scale API pods down, rather than paying for peak capacity around the clock.
